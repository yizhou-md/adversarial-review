# adversarial-review

[English](README.md) | [简体中文](README-zh.md)

> 交付前，让另一个脑子专门找反例。

`adversarial-review` 是一个 Codex 技能。它会在你真正信任一份结果之前，先挑战它：代码、计划、diff、拉取请求、实现结果、agent 输出都可以。它适合那种“看起来没问题”还不够的场景：你真正想知道的是，什么证据会证明这件事其实没有做好？

## 它解决什么问题

普通自检常常默认“我大概是对的”。对抗式评审把这个默认反过来：先把结果写成可以被推翻的被测主张，再主动寻找会推翻它的反例。

这不是为了把语气变得刻薄，也不是多跑一次 lint。它关心的是那些会改变结论的具体场景：边界输入、缺失验证、错误权限假设、过度设计，或者不同评审者没有看同一份材料。

## 它是怎么工作的

`adversarial-review` 把证伪过程拆成三步：

1. **建立假设**：把“我觉得没问题”转成一个可被反例推翻的被测主张，并写清成功标准和失败条件。
2. **独立找反例**：冻结评审范围，让评审者或多个评审轮次从指定视角寻找具体反例。
3. **判定结论**：判断发现是否改变最终结论，返回 `PASS`、`CONTESTED` 或 `REJECT`。

和普通自检相比，它会明确记录评审是否足够独立，但不会要求你先记住一套内部术语。评审者可以按你指定的评审路径运行，也可以按默认策略选择：例如由 Codex 调用 Claude Code、使用同一工具里的 subagent、启动全新的同工具 CLI 会话，或明确标注为单 agent 手动评审。

这个技能参考了公开的 [`pedronauck/skills` `adversarial-review` skill](https://claudemarketplaces.com/skills/pedronauck/skills/adversarial-review)。它保留了好用的评审视角模式，并针对 Codex 调整了评审路径的选择规则：你可以自己指定路径；没有指定时，技能按上下文处理；它会记录真实的独立性状态，但不会因为某条路径刚好可用就替你排序或擅自切换。

## 你会得到什么

- 一段清楚的评审意图，以及一个可以被反例推翻的被测主张。
- 一个冻结后的评审范围，确保所有评审者看的是同一份材料。
- 来自一个或多个视角的发现：`Skeptic`、`Architect` 和 `Minimalist`。
- 一个综合判断：`PASS`、`CONTESTED` 或 `REJECT`。
- 一组可追踪的评审记录：哪里可能错、为什么错、是否影响结论，以及下一步怎么处理。

## 目录

- `skills/adversarial-review/SKILL.md`：评审工作流、评审视角、提示词模板、综合规则和结论格式。

## 安装

使用 Agent Skills CLI 安装：

```sh
pnpx skills add https://github.com/yizhou-md/adversarial-review --skill adversarial-review
```

## 使用

告诉 agent 你要评审什么即可。你可以指定评审路径，但不需要在使用前先弄懂全部内部术语：

```text
Use $adversarial-review to review this diff with a Codex subagent and Claude Code.
```

```text
请用 $adversarial-review 审阅当前 diff，优先使用 Codex subagent。
```

这个技能会：

1. 把你的意图转成一个可证伪的被测主张，并写清成功标准和失败条件。
2. 冻结评审范围：diff、PR、计划、文件，或其他具体材料。
3. 解析评审路径：使用你指定的路径；交互式聊天里没有指定时，列出选项并询问；非交互式自动化中默认使用同工具 subagent。
4. 在所选路径支持时，按指定评审视角分派评审者。
5. 合并重复发现，并判断每条发现是否会改变最终结论。
6. 返回包含具体证据、失败场景和下一步建议的评审结论。

## 评审视角

| 视角 | 关注点 |
| --- | --- |
| `Skeptic` | 检查工作是否正确、完整，并且已经被充分验证。 |
| `Architect` | 检查结构、边界和契约是否服务于既定目标。 |
| `Minimalist` | 检查实现是否可以更简单、更小、更少猜测，避免没有证据支撑的复杂度。 |

## 评审路径（谁来评审）

你可以指定要使用哪种评审路径，也可以让“未指定评审路径时的策略”自动生效。这里的“路径”指评审者的执行方式，不是文件路径；技能不会仅仅因为某条路径可用，就擅自替你选择它。

| 路径 | 示例 |
| --- | --- |
| 跨工具评审者 | Codex 调用 Claude Code，或 Claude Code 调用 Codex |
| 同工具 subagent | Codex 启动隔离的 Codex subagents |
| 同工具 CLI 会话 | Codex 启动一个新的 `codex exec` 评审者 |
| 单 agent 手动评审 | 主 agent 自己运行每个评审视角，并标注没有独立评审者 |

如果交互式聊天里没有指定路径，技能会列出评审路径选项，建议使用同工具 subagent，并在分派前询问用户。在非交互式自动化场景中，默认路径是同工具 subagent。如果指定路径或默认路径不可用，技能会报告阻塞原因，而不是静默切换到另一条路径。

默认情况下，评审工具会加载正常的用户配置和项目配置，让本地规则、团队政策和项目约定在评审期间继续生效。评审者默认应保持只读；如果某条路径无法可靠限制权限，就传入静态评审包，而不是开放 workspace 访问权限。

独立性和权限状态是两件事。跨工具评审可能更独立，但不一定拥有文件系统只读沙箱；工具限制不等于文件系统沙箱。因此，技能会记录真实权限状态，而不是在结论里把它夸大成更强的隔离。对 Claude Code 来说，静态评审包加禁用工具应记录为 `static packet / tools disabled`；通过 plan mode 或工具权限控制访问 live workspace 时，应记录为 `tool-restricted plan mode`，或记录实际使用的权限状态。

## 何时不使用

- 不要把这个技能用于普通快速自检；用户没有要求对抗式评审时，不应主动启动它。
- 不要把默认路径当成安装缺失工具或静默切换路径的许可。
- 不要把它当作编辑或修复流程；除非用户在评审后明确要求修复，否则这个技能只产出评审结论。
- 不要把它用于只涉及风格、格式化或 lint 清理的问题；这类任务通常不需要对抗式风险评审。

## 验证

这个仓库包含一个轻量级验证脚本，用来检查技能包结构、评审路径策略、文档和核心工作流约束；它不会真的分派 reviewer 工具。

```sh
./runtest.sh
```
